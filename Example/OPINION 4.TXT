### 1. 해당 확장 프로그램의 접근 시스템?을 바꿀 예정입니다. 
<예시자료: Example/Worldinfoinfo>
변경 후 동작: 해당 확장을 다운받으면 기존과 같이 📱 핸드폰 이모지 아이콘이 추가됩니다. (*즉, 퀵센드나 삭제된 메세지입니다 매크로 버튼은 rightsendform에서 삭제하고, 오직 📱 핸드폰 이모지 아이콘만 남습니다.)
해당 이모지를 누르면 팝업이 등장합니다. 맨 위에 주 패널창으로 들어가는 버튼이 있고, 그 밑으로 내가 퀵 액세스로 지정한 기능들이 쭈르륵 뜹니다.

예시를 들어보겠습니다. 제가 퀵 액세스로 유저 메세지로 이미지 생성하기 기능, AI 트리거 없이 메세지를 전송하는 퀵센드 기능, 읽씹 기능, 연락 안 됨 기능, SNS 게시글 조회 기능, 통화 기능을 등록했다고 칩시다.
----------------------------
📱 메인패널로 이동 (순서 1번으로 고정)
🎨 유저 이미지 전송
📞 통화 요청
🔕 읽씹하기
📸 SNS 들어가기
💌 트리거 없이 메세지 전송
---------------------------
아이콘을 클릭했을 때 이런 식으로 나옵니다.
유저 이미지 전송 버튼을 누르면 이미지 전송을 바로 할 수 있도록 입력창이 뜨고, 통화 요청을 누르면 바로 통화 요청이 가고, (통화 패널 열기 X 그것과 별개입니다.) 읽씹하기를 누르면 읽씹한 후의 답변이 바로 트리거되고, SNS 들어가기를 누르면 바로 SNS 게시글을 조회할 수 있게 되고, 트리거 없이 메세지 전송을 누르면 quick send 기능을 사용합니다.

즉, 이를 위해선 기능을 세분화하고, 팝업에 링크(연동)하는 기능이 필요할 것입니다. 어떤 기능을 연동할지는 설정-퀵액세스 탭에서 설정할 수 있습니다. 단 기능을 세분화하여 나누고 팝업에 각각 연동시킬 수 있도록 퀵 액세스 탭 기능, 나아가 전반적인 구조 및 기능 상세 분류화, 그리고 리팩토링이 필요할 것입니다.

### 2. 이미지 프롬프트 주입의 개선
A. 연락처에 등록된 캐릭터의 이름을 기반으로 외모태그를 트래킹하여 주입합니다.
설명: 현재 연락처에 user(외모태그: girl, white hair)와 A (외모태그: boy, black hair)라는 캐릭터가 등록되어있다고 가정합시다.
user가 이미지를 생성하여 보낼 때 user와 A가 셀카를 찍었다는 컨셉으로 인풋을 작성했습니다. 그러나 현재 로직에서 image api에 주입되는 것은 user의 외모태그 뿐입니다. 즉, 생성된 결과물은 user(girl, white hair)와 완전히 랜덤생성된 캐릭터라는 것입니다. user의 외모태그만 들어가기 때문에 퍼포먼스가 떨어지기 때문에, 인풋을 등록할 때 연락처에 등록된 이름을 트래킹하여 외모태그를 가져옵니다.
부가설명: user를 제외하고 연락처에 a(boy, white hair)와 b(dog, blonde fur) 가 있을때, user가 'user와 a의 셀카'라고 하면 user의 외모태그와 a의 외모태그를 같이 활성화합니다. 'a와 b의 셀카'라고 할 경우 a와 b의 외모태그를 활성화합니다. 즉 user가 인풋에 '엉망인 주방의 모습'을 기재했다면, 그 어떤 외모태그도 불러오지 않습니다.

B. image api에 주입하는 프롬프트를 개선합니다.
현재 image api에 주입되는 프롬프트는 다음과 같습니다.

[공통 프롬프트 접두사, 태그 생성ai가 만든 태그, 외모 태그]

그러나 이러면 2명의 캐릭터가 이미지생성에 들어갈시 제대로 구분되지 않습니다. 때문에 아래와 같이 프롬프트를 개선합니다.

[ 공통 프롬프트 접두사, 태그 생성 ai가 만든 태그 | 외모태그 1 | 외모 태그 2 | 외모 태그 3... ]
이렇게 하면 image api가 헷갈리지 않을 수 있습니다. (즉, 전체 프롬프트와 외모태그를 |를 통해 나눠주는 작업입니다.) 여기서 외모태그 1, 2, 3은 유동적이며 경우에 따라 있을 수도 없을 수도 있습니다. (사유: A에 기재한대로, 연락처에 적힌 이름을 캐치하여 외모태그를 주입하기 때문입니다.)

C. char가 이미지를 먼저 생성하여 보내는 로직 강화
- 컨텍스트를 더욱 면밀하게 살펴 '사진' 이라는 직접적인 얘기가 없을지라도, 컨텍스트상 적절한 타이밍이면 보낼 수 있도록 ai에게 자체 thinking을 부여합니다. (예시: 싸늘한 무표정의 char. user가 브이해주면 안돼? ㅠㅠ 함. 이 경우 char가 브이하는 사진을 이미지생성하여 보냅니다.)
- 현재 빈번하진 않지만 종종 생기는 오류가 있습니다: 첫 이미지 생성 이후 추가로 이미지를 요구했을 경우 실제로 이미지 생성이 이루어지지 않고, 기존 컨텍스트에 포함된 이미지 링크의 내용만 변환하여 다시보내는 오류. 즉 해당 로직에 허점 또는 오류가 있는 것 같기에 이 부분을 검토하여 보강해주셨으면 합니다.

D. 기본적으로 모든 이미지 생성api 요청 양식을 [ 공통 프롬프트 접두사, 태그 생성 ai가 만든 태그 | 외모태그 1 | 외모 태그 2 | 외모 태그 3... ]로 합니다. 즉, 모든 이미지 생성은 메세지생성이든 sns생성이든 태그생성ai가 사전검수를 거쳐 생성해낸 태그여야 합니다.

E. SNS 이미지 프롬프트 / 메신저 이미지 프롬프트를 개선합니다. danbooru 태그를 기준으로 상황을 분석하여 적절한 조합으로 생성해야 하며 _(언더바) 대신 띄어쓰기를 사용하라는 지침을 추가합니다.

### 3. SNS 프로필 연동 수정

- 현재 char의 프로필이 자동 연동이 되지 않습니다. user와 char는 자동연동 기능이 꺼져있더라도 무조건 연동되어야 합니다. 이 오류를 수정합니다.

### 4. SNS 게시글 생성로직 관련

- ai가 이미지 생성을 하였을 경우, '사진설명' 항목은 기입하지 않도록 수정합니다.
- 최소 좋아요 수 설정: SNS 프로필 설정에서 최소 좋아요 수를 설정할 경우, 좋아요 수는 해당 수보다 작아지지 않도록 수정하며, 최소 좋아요 수보다 많도록 랜덤 숫자를 출력합니다. 


### 5. 마치며

1번의 요청사항의 경우 오류가 생길 여지가 매우 많아질 것이라 예상됩니다. 오류가 나지 않도록 더욱 면밀한 검수 및 리팩토링 진행을 부탁드립니다.
